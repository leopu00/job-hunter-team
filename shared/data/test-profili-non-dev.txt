ANALISI STRUTTURALE — candidate.skills INADEGUATA PER NON-DEV
══════════════════════════════════════════════════════════════════════

La struttura candidate.skills nell'example e' pensata SOLO per developer:
  {languages, frameworks, databases, tools, cloud}

Per profili non-dev, questo schema e' semanticamente inadeguato:

PROFILO          | FORZATURA SEMANTICA
-----------------|-------------------------------------------------
Marketing (A)    | IMPOSSIBILE mappare: usa struttura custom
                 | primary/secondary/learning → incompatibile
Product Mgr (B)  | SQL in 'languages': accettabile
                 | Agile/Scrum in 'frameworks': improprio
                 | Amplitude/Mixpanel in 'databases': sbagliato
Data Analyst (C) | SQL/R in 'languages': parzialmente ok
                 | Statistical Analysis in 'frameworks': sbagliato
                 | Excel in 'databases': sbagliato semanticamente
                 | cloud: [] sempre vuoto

CONSEGUENZA:
  - Un marketing manager non riesce a compilare la sezione senza
    forzature o struttura custom incompatibile
  - Lo scrittore riceve dati semanticamente sbagliati e potrebbe
    generare CV con categorizzazioni errate (es. "Excel" nella
    sezione "Databases" del CV)

SOLUZIONE PROPOSTA — Struttura universale flessibile:

  candidate:
    skills:
      # Per developer: usa languages/frameworks/databases/tools/cloud
      # Per non-dev: usa primary/secondary/tools (ignora il resto)
      # Struttura OPZIONALE — usa solo le categorie rilevanti per te
      primary: []       # skill principali (qualsiasi ruolo)
      secondary: []     # skill secondarie (qualsiasi ruolo)
      tools: []         # tool e software (tutti i ruoli)
      # Categorie opzionali solo per developer:
      languages: []     # linguaggi di programmazione
      frameworks: []    # framework e librerie
      databases: []     # database e storage
      cloud: []         # cloud provider e servizi

  Con questa struttura:
  - Lo scrittore usa primary+secondary+tools come base universale
  - Per dev, usa anche languages+frameworks+databases+cloud
  - Nessun KeyError per nessun tipo di profilo
